# Блок 1. Тестирование требований, бизнес-процесс ЭТрН

Чек-лист и тест-кейсы построены по предоставленной BPMN-диаграмме процесса (дорожки: Грузоотправитель, Оператор ЭДО, Перевозчик, Водитель, Грузополучатель) и действующему формату ФНС (приказ № ЕД-7-26/1065@). Обмен идёт через ИС ЭПД и далее в ГИС ЭПД, подписание через КЭП. Места, которые надо сверить с XSD-схемой, помечены так: [свериться с XSD].

## 1.1. Процесс по BPMN (как он читается на диаграмме)

Документ формируется последовательно четырьмя титулами. Каждый титул подписывается через КЭП и маршрутизируется Оператором ЭДО.

| Титул | Кто формирует | Содержание | Статус ЭТрН после |
|---|---|---|---|
| Титул 1 | Грузоотправитель | Условия договора, сведения о сторонах сделки, водителе, грузе, сопроводительных бумагах | "Ожидает отгрузки со склада" |
| Титул 2 | Перевозчик | Подтверждение получения, данные о погрузке, замечания при погрузке (если есть), генерация QR-кода водителю | "Груз принят к перевозке" |
| Титул 3 | Грузополучатель | Приём транспорта, решение "принять груз", данные о приёмке | "Груз принят грузополучателем" |
| Титул 4 | Перевозчик | Итоги разгрузки, замечания при выгрузке (если есть) | "Завершён" |

Поток по happy path:

1. Грузоотправитель: создать ЭТрН, заполнить обязательные сведения титула 1, сохранить черновик, подписать и отправить титул 1, статус становится "Ожидает отгрузки со склада".
2. Оператор ЭДО: получает титул 1, делает маршрутизацию и согласование, рассылает участникам.
3. Перевозчик: подтверждение о получении, получает информацию о погрузке, дальше развилка "Есть замечания?". Если да, активирует чекбокс "Замечания при погрузке" и вносит несоответствия. Если нет, идёт дальше. Затем подписывает и отправляет титул 2 и генерирует QR-код водителю.
4. Водитель: контроль погрузки, передаёт данные о погрузке диспетчеру, по QR-коду прибывает на место разгрузки.
5. Оператор ЭДО: рассылает второй титул грузоотправителю и грузополучателю. Грузоотправитель ставит статус "Груз принят к перевозке".
6. Грузополучатель: получает титул 2, ждёт ТС, принимает транспорт на разгрузку, дальше развилка "Принять груз?". Если нет, ветка "Отказ от груза" и процесс завершается. Если да, активирует чекбокс "Груз принят", заполняет данные о приёмке, подписывает и отправляет титул 3.
7. Оператор ЭДО: рассылает третий титул грузоотправителю и перевозчику и отправляет третий титул в ГИС ЭПД. Грузоотправитель ставит статус "Груз принят грузополучателем".
8. Перевозчик: дожидается разгрузки, докладывает об итогах, дальше развилка "Замечания при выгрузке?". Если да, активирует чекбокс и вносит несоответствия. Если нет, подписывает и отправляет титул 4.
9. Оператор ЭДО: рассылает четвёртый титул грузоотправителю и грузополучателю. Грузоотправитель ставит статус "Завершён".

## 1.2. Чек-лист основного бизнес-процесса

### Титул 1, создание (Грузоотправитель)
- [ ] Действие "Создать ЭТрН" доступно грузоотправителю, документ создаётся в начальном состоянии.
- [ ] Обязательные сведения титула 1 (стороны сделки, водитель, груз, сопроводительные бумаги) можно заполнить, и они обязательны.
- [ ] "Сохранить версию как черновик" сохраняет без подписи, черновик не уходит Оператору ЭДО, данные не теряются при перезагрузке.
- [ ] "Подписать и отправить титул" доступно только при заполненных обязательных полях и валидной КЭП.
- [ ] После подписи титула 1 статус становится "Ожидает отгрузки со склада", разделы титула 1 становятся неизменяемыми.

### Маршрутизация (Оператор ЭДО)
- [ ] Подписанный титул 1 принимается Оператором ЭДО, проходит маршрутизацию и согласование, рассылается участникам.
- [ ] Перевозчик и водитель получают свои уведомления и данные.

### Титул 2, погрузка (Перевозчик и Водитель)
- [ ] Перевозчик подтверждает получение и получает информацию о погрузке.
- [ ] Развилка "Есть замечания?": при "да" активируется чекбокс "Замечания при погрузке" и доступно поле несоответствий, при "нет" процесс идёт дальше без этих полей.
- [ ] Подписание титула 2 валидной КЭП приводит к генерации QR-кода водителю.
- [ ] Водитель проходит "Контроль погрузки" и "Передать данные о погрузке диспетчеру", QR-код принимается на стороне водителя.
- [ ] После рассылки титула 2 грузоотправитель ставит статус "Груз принят к перевозке".

### Титул 3, приёмка (Грузополучатель)
- [ ] Грузополучатель получает титул 2, ждёт ТС, принимает транспорт на разгрузку.
- [ ] Развилка "Принять груз?": "нет" ведёт в ветку "Отказ от груза", которая корректно завершается (см. негативные и граничные кейсы), "да" даёт чекбокс "Груз принят" и поля данных о приёмке.
- [ ] Подписание титула 3 валидной КЭП.
- [ ] Оператор ЭДО рассылает титул 3 и отправляет его в ГИС ЭПД, приходит положительный квиток.
- [ ] Грузоотправитель ставит статус "Груз принят грузополучателем".

### Титул 4, разгрузка (Перевозчик)
- [ ] Перевозчик дожидается разгрузки и докладывает об итогах.
- [ ] Развилка "Замечания при выгрузке?": "да" даёт чекбокс и несоответствия, "нет" ведёт дальше.
- [ ] Подписание титула 4, рассылка грузоотправителю и грузополучателю, статус "Завершён".

### Сквозные проверки
- [ ] Каждая смена статуса логируется (участник, метка времени).
- [ ] Каждый титул подписывается КЭП своего участника, чужие титулы недоступны для редактирования и подписи.
- [ ] Отправка в ГИС ЭПД (титул 3): ошибка или недоступность ГИС обрабатывается, документ не теряется, возможен повтор.

## 1.3. Ключевые проверки формы и документа (по XSD) [свериться с XSD]

- Обязательные и опциональные элементы каждого титула. Документ без обязательного элемента не проходит схемную валидацию.
- Кардинальность повторяющихся блоков (грузы, места): N вхождений там, где их несколько, и ровно одно там, где элемент единичный.
- Типы и форматы: ИНН (10 или 12 знаков плюс контрольное число), госномер ТС, даты, числовые поля (вес, объём, суммы) с допустимой точностью, справочные поля только из перечислений.
- Длины строк: ровно максимум принимается, максимум плюс один отклоняется.
- Подписанный XML: изменение любого байта данных делает подпись КЭП невалидной.

## 1.4. Негативные и граничные проверки с ожидаемым поведением

| # | Сценарий | Ожидаемое поведение |
|---|---|---|
| N1 | Подписание любого титула отозванной или просроченной КЭП | Отказ, статус не меняется, в ГИС ЭПД ничего не уходит |
| N2 | Перевозчик пытается подписать титул 2 до подписи титула 1 грузоотправителем | Действие недоступно, нарушение порядка титулов блокируется |
| N3 | Ветка "Отказ от груза": грузополучатель выбирает "Принять груз: нет" | Процесс завершается ветвью отказа, титул 3 о приёмке не формируется, статусы грузоотправителя и перевозчика отражают отказ, а не "Груз принят грузополучателем" |
| N4 | "Есть замечания: да", но поле несоответствий пустое | Подписать титул 2 нельзя: при активном чекбоксе замечаний поле обязательно |
| N5 | Грузополучатель пытается отредактировать или подписать титул 2 (чужой) | Запрещено, грузополучатель работает только с титулом 3 |
| N6 | Повторная отправка титула 1 после уже установленного статуса "Ожидает отгрузки со склада" | Дубль отклоняется, статус и история не дублируются |
| N7 | Недоступность ГИС ЭПД при отправке титула 3 | Статус "ошибка передачи", корректный повтор без дублей в ГИС ЭПД |
| N8 | Попытка поставить статус "Завершён" до подписания титула 4 | Запрещено, статус "Завершён" только после рассылки титула 4 |
| N9 | Замечания при выгрузке: "да", поле несоответствий пустое | Подпись титула 4 блокируется до заполнения |
| N10 | QR-код использован, просрочен или предъявлен повторно при прибытии водителя на разгрузку | Прибытие на разгрузку не подтверждается по невалидному QR, выдаётся понятная ошибка |

## 1.5. Баг-репорты

### BUG-1
Заголовок: Грузоотправитель ставит статус "Груз принят грузополучателем", когда грузополучатель выбрал ветку "Отказ от груза"

Серьёзность: Critical. Приоритет: High.

Предусловия: ЭТрН в статусе "Груз принят к перевозке" (титул 2 подписан). На развилке "Принять груз?" грузополучатель выбирает "нет" (Отказ от груза).

Шаги воспроизведения:
1. Грузополучатель принимает транспорт на разгрузку.
2. На развилке "Принять груз?" выбирает "нет", уходит в ветку "Отказ от груза".
3. Проверить статус ЭТрН на стороне грузоотправителя.

Фактический результат: грузоотправителю доступен или автоматически выставляется статус "Груз принят грузополучателем", при этом титул 3 о приёмке отсутствует.

Ожидаемый результат: при отказе от груза статус "Груз принят грузополучателем" недоступен, процесс идёт по ветке отказа, статусы участников отражают отказ. Расхождение статуса и факта приёмки недопустимо, документ это первичка для учёта.

### BUG-2
Заголовок: Титул 2 подписывается с активным чекбоксом "Замечания при погрузке", но с пустым полем несоответствий

Серьёзность: Major. Приоритет: Medium.

Предусловия: перевозчик на этапе погрузки, развилка "Есть замечания: да".

Шаги воспроизведения:
1. Перевозчик активирует чекбокс "Замечания при погрузке".
2. Поле "несоответствия" оставляет пустым.
3. Нажимает "Подписать и отправить титул 2".

Фактический результат: титул 2 подписывается и уходит Оператору ЭДО с активным признаком замечаний, но без их описания.

Ожидаемый результат: при активном чекбоксе замечаний поле несоответствий обязательно, подпись блокируется с сообщением до заполнения. Та же логика нужна для замечаний при выгрузке в титуле 4.

## 1.6. Критерии оценки, самопроверка

Полнота. Покрыт happy path по четырём титулам, обе развилки (замечания при погрузке и выгрузке, принять или отказ от груза), отправка в ГИС ЭПД, граничные кейсы (КЭП, порядок титулов, QR-код, длины и типы по XSD).

Понимание процесса. Проверки идут по ролям и последовательности титулов из BPMN и по статусной модели, а не по формальному перебору полей.

Качество баг-репортов. Воспроизводимые шаги, привязка к конкретным узлам BPMN и к бизнес-последствиям, разделены severity и priority.
