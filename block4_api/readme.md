# Блок 4. Тестирование API, контроллер /api/v2/orders

Чек-лист построен строго по предоставленному описанию контроллера (api.md) и индустриальным гайдам (Google API Design Guide, Microsoft REST API Guidelines). Ключевые особенности контракта: идентификатор это UUID; удаление логическое (DELETE) с парным восстановлением (PUT /.../undelete); из-за этого для удалённых записей возвращается 410 Gone; POST поддержан на двух URL; HEAD используется для проверки существования.

## Контракт (из api.md)

| Метод | Назначение | URL | Ожидаемые коды |
|---|---|---|---|
| GET | List | /api/v2/orders | 200 |
| GET | Get | /api/v2/orders/{id:uuid} | 200, 404, 410 |
| POST | Create | /api/v2/orders/ или /api/v2/orders/{id:uuid} | 200/201, 409 |
| PUT | Update | /api/v2/orders/{id:uuid} | 200/204 |
| PUT | Create or Update | /api/v2/orders/{id:uuid} | 201/200 при создании, 204/200 при обновлении |
| DELETE | Delete (логическое) | /api/v2/orders/{id:uuid} | 204, 404, 410 |
| HEAD | - | /api/v2/orders/{id:uuid} | 204 |

## 4.1. GET List, /api/v2/orders

| # | Сценарий | Ожидаемо |
|---|---|---|
| L1 | Есть записи | 200 OK, массив T[] с элементами |
| L2 | Записей нет | 200 OK и пустой массив [], а не 404 и не null |
| L3 | В выборке есть логически удалённые записи | Они не возвращаются, только не удалённые |
| L4 | После DELETE одной записи повторный List | Удалённой записи в выдаче нет |
| L5 | Формат ответа | Тип элементов соответствует read-model контракта |

## 4.2. GET Get, /api/v2/orders/{id:uuid}

| # | Сценарий | Ожидаемо |
|---|---|---|
| G1 | Существующая активная запись | 200 OK и ресурс |
| G2 | UUID валидный, записи никогда не было | 404 Not Found |
| G3 | Запись была логически удалена через DELETE | 410 Gone, а не 404, это ключевое отличие контракта |
| G4 | Запись удалена, затем восстановлена через PUT /undelete | 200 OK, снова доступна |
| G5 | id не UUID (например 123 или abc) | 400 Bad Request, невалидный формат маршрута [свериться: 400 или 404] |
| G6 | UUID правильного формата, но нулевой или служебный | 404 или 410 по состоянию |

## 4.3. POST Create, /api/v2/orders/ и /api/v2/orders/{id:uuid}

| # | Сценарий | Ожидаемо |
|---|---|---|
| C1 | POST /orders/ с валидным телом, id генерит сервер | 201 Created (есть парный GET /{id}) и заголовок Location на новый ресурс |
| C2 | POST /orders/{uuid}, id ещё не существует | 201 Created, создание по заданному id |
| C3 | POST /orders/{uuid}, id уже существует | 409 Conflict, повторно создать по тому же id нельзя |
| C4 | POST /orders/{uuid}, id ранее логически удалён | Определить поведение: 409 (id занят) или 410, либо восстановление [свериться] |
| C5 | POST без обязательного поля | 400 Bad Request, структурированная ошибка |
| C6 | POST с неверным типом поля | 400 |
| C7 | Битый JSON | 400, а не 500 |
| C8 | Content-Type не application/json | 415 Unsupported Media Type |
| C9 | Тело превышает лимит размера | 413 Payload Too Large [свериться] |
| C10 | Различие 200 и 201 | 201 именно при наличии парного GET /{id}, по контракту |

## 4.4. PUT, Update и Create-or-Update, /api/v2/orders/{id:uuid}

| # | Сценарий | Ожидаемо |
|---|---|---|
| U1 | PUT существующей записи, сервер возвращает read-model | 200 OK и обновлённый ресурс |
| U2 | PUT существующей записи без возврата тела | 204 No Content |
| U3 | PUT (Create-or-Update), id не существовал, создание | 201 Created (предпочтительно, чтобы различать сценарий) или 200 |
| U4 | PUT (Create-or-Update), id существовал, обновление | 204 No Content (предпочтительно) или 200 |
| U5 | Идемпотентность: повторный одинаковый PUT | Тот же результат, без побочных эффектов |
| U6 | PUT логически удалённой записи | Определить семантику: 410 либо восстановление с обновлением [свериться] |
| U7 | PUT с невалидным телом | 400 |
| U8 | Частичное тело в PUT (PUT это полная замена) | Отсутствующие поля обнуляются и сбрасываются, а не сохраняются, это семантика PUT, а не PATCH |
| U9 | Различение 201 и 204 | По контракту коды должны явно показывать "создал" против "обновил" |

## 4.5. DELETE (логическое), /api/v2/orders/{id:uuid}

| # | Сценарий | Ожидаемо |
|---|---|---|
| D1 | DELETE существующей активной записи | 204 No Content, запись помечена удалённой |
| D2 | GET после DELETE | 410 Gone |
| D3 | List после DELETE | Записи в выдаче нет |
| D4 | Повторный DELETE уже удалённой | 410 Gone, запись логически удалена |
| D5 | DELETE несуществующего id | 404 Not Found |
| D6 | DELETE с невалидным UUID | 400 или 404 [свериться] |
| D7 | Восстановление PUT /orders/{id}/undelete после DELETE | Запись снова активна, GET даёт 200 |
| D8 | undelete записи, которая не была удалена | Определить: 409, 404 или идемпотентный 200 [свериться] |

## 4.6. HEAD, /api/v2/orders/{id:uuid}

| # | Сценарий | Ожидаемо |
|---|---|---|
| HD1 | Существующая запись | 204 No Content, без тела, заголовки как у GET |
| HD2 | Несуществующая | 404 [свериться: контракт декларирует 204, уточнить поведение для отсутствующих] |
| HD3 | Логически удалённая | 410 [свериться] |
| HD4 | Тело ответа на HEAD | Всегда пустое, даже при 4xx |

## 4.7. Авторизация и доступ

| # | Сценарий | Ожидаемо |
|---|---|---|
| A1 | Запрос без токена | 401 Unauthorized и заголовок WWW-Authenticate |
| A2 | Просроченный или невалидный токен | 401 |
| A3 | Валидный токен, но нет прав на операцию | 403 Forbidden, отличать от 401 |
| A4 | Доступ к чужому заказу | 403 или 404 [свериться: скрывать ли существование] |

## 4.8. Падение БД и серверные ошибки

| # | Сценарий | Ожидаемо |
|---|---|---|
| S1 | БД недоступна на GET, POST или PUT | 503 Service Unavailable (предпочтительно) или 500 |
| S2 | Таймаут БД | 503 или 504, при необходимости заголовок Retry-After [свериться] |
| S3 | Необработанное исключение | 500, тело это обобщённое сообщение и trace или correlation id, без стектрейса и SQL |
| S4 | Сбой во время POST или PUT, транзакция не закоммитилась | Атомарность: запись либо создана и обновлена, либо нет, без полусозданных заказов |
| S5 | Сбой между DELETE и записью флага | Состояние консистентно, либо удалена, либо нет |

## 4.9. Кросс-функциональное (Google и MS)

- Метод не разрешён на ресурсе (например PATCH /orders/{id}, если он не поддержан): 405 Method Not Allowed и заголовок Allow.
- Единый формат ошибок для всех 4xx и 5xx (error.code, error.message, error.status).
- 410 Gone это отличительная черта контракта. Во всех методах над логически удалённым ресурсом проверяем именно 410, а не 404.
- Версия в пути /v2/: обращение к несуществующей версии должно иметь определённое поведение.
- Идемпотентность: GET, PUT, DELETE, HEAD идемпотентны, POST нет (повторный POST /{id} даёт 409).
- Безопасность: UUID снижает перечислимость, проверяем, что 404, 410 и 403 не раскрывают лишнего о чужих записях.
