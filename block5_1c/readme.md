# Блок 5. Базовые вопросы про 1С и структуру обменов

## Часть А. Запрос актуальной стоимости доставки

Регистр сведений "Цены доставки" хранит стоимость по измерениям: Период, ТипКлиента и интервал веса (ВесОт, ВесДо). По параметрам (Дата заказа, Тип клиента, Общий вес) надо вернуть актуальную цену с учётом трёх вещей:

1. берём запись с максимальной датой начала, которая меньше или равна дате заказа (то есть срез на дату, а не "на сейчас");
2. границы веса без перекрытия: вес = 10 должен попасть в интервал, который заканчивается на 10, а не в тот, что с 10 начинается;
3. если вес больше максимального известного интервала, стоимость не определена, возвращаем NULL или 0.

### Запрос на языке запросов 1С

```bsl
ВЫБРАТЬ ПЕРВЫЕ 1
    ЦеныДоставкиСрезПоследних.Стоимость КАК Стоимость
ИЗ
    РегистрСведений.ЦеныДоставки.СрезПоследних(
        &ДатаЗаказа,
        ТипКлиента = &ТипКлиента
            И ВесОт < &ОбщийВес
            И ВесДо >= &ОбщийВес
    ) КАК ЦеныДоставкиСрезПоследних
УПОРЯДОЧИТЬ ПО
    ЦеныДоставкиСрезПоследних.ВесДо ВОЗР
```

Обработка результата, чтобы корректно отработать случай "вес вне интервалов":

```bsl
Результат = Запрос.Выполнить();
Если Результат.Пустой() Тогда
    Возврат NULL; // или 0, по требованию: стоимость не определена
КонецЕсли;
Выборка = Результат.Выбрать();
Выборка.Следующий();
Возврат Выборка.Стоимость;
```

### Почему так

Срез последних `СрезПоследних(&ДатаЗаказа, Условие)` это штатный механизм периодического регистра сведений. Для каждой комбинации измерений он вернёт последнюю запись с периодом не больше даты заказа. Это и есть требование "максимальная дата начала, меньшая или равная дате заказа", без ручного `MAX(Период)` и подзапросов.

Границы веса `ВесОт < Вес И ВесДо >= Вес` дают полуоткрытый интервал вида (ВесОт; ВесДо]. Тогда вес = 10:

* не попадёт в интервал [10; 20], заданный как ВесОт = 10 (потому что 10 < 10 ложно);
* попадёт в интервал ...; 10] с ВесДо = 10 (потому что 10 >= 10 истинно).

Так убираем перекрытие на стыке, как и просили в задании. Если в данных принята обратная конвенция [ВесОт; ВесДо), то знаки зеркалятся, и эту конвенцию надо зафиксировать с аналитиком.

`ВЫБРАТЬ ПЕРВЫЕ 1` вместе с `УПОРЯДОЧИТЬ ПО ВесДо` это страховка от пересекающихся интервалов в данных: берём самый узкий подходящий. Если данные корректные и интервалы не пересекаются, подойдёт ровно одна строка.

Вес вне интервалов даёт пустой результат среза, и мы возвращаем NULL или 0, как требует условие.

Маленькое уточнение: параметр `&ОбщийВес` физически передаётся через `Запрос.УстановитьПараметр(...)`. В тексте запроса условие по весу это отбор внутри виртуальной таблицы среза, а не `ГДЕ` снаружи. Иначе срез сначала свернёт данные по измерениям, и условие по весу отработает не так, как нужно.

## Часть Б. Разбор метода РассчитатьСтоимость, ошибки и проблемы

Нашёл 8 проблем (требовалось не меньше 5). По каждой коротко суть и как починить.

### 1. Логика МЕЖДУ на границах неверна

В запросе стоит `&СуммаВес МЕЖДУ ЦеныДоставки.ВесОт И ЦеныДоставки.ВесДо`. Оператор `МЕЖДУ` в 1С включает обе границы (ВесОт <= Вес <= ВесДо). Из-за этого появляется перекрытие на стыках: вес = 10 попадёт сразу и в [0;10], и в [10;20], то есть две строки и недетерминированный результат. А задание прямо требует, чтобы перекрытия не было.

Как починить: заменить на `ЦеныДоставки.ВесОт < &СуммаВес И ЦеныДоставки.ВесДо >= &СуммаВес` (полуоткрытый интервал).

### 2. Не используется срез последних

Условие `ГДЕ ... ЦеныДоставки.Период <= &ДатаЗаказа` вернёт вообще все исторические записи до даты заказа, а не только актуальную. Сортировка `УПОРЯДОЧИТЬ ПО Период УБЫВ` плюс взятие первой строки частично спасает, но при нескольких типах клиента и интервалах веса сортировка только по периоду не гарантирует, что первая строка это нужная комбинация.

Как починить: использовать виртуальную таблицу `РегистрСведений.ЦеныДоставки.СрезПоследних(&ДатаЗаказа, <условия>)`.

### 3. Нет ВЫБРАТЬ ПЕРВЫЕ 1

Код берёт первую строку выборки вручную (`Выборка.Следующий()`), но сам запрос тянет все подходящие строки. При перекрытии интервалов (см. пункт 1) "первая" строка зависит от порядка, отсюда недетерминизм.

Как починить: добавить `ВЫБРАТЬ ПЕРВЫЕ 1` и явную сортировку, которая однозначно определяет выбор.

### 4. Неправильное обращение СправочникКлиенты.НайтиПоРеквизиту(...)

Тут две проблемы. Во-первых, у менеджера справочника нет универсального метода `НайтиПоРеквизиту` с таким синтаксисом (есть `НайтиПоНаименованию`, `НайтиПоКоду`, есть `НайтиПоРеквизиту(ИмяРеквизита, Значение)`, но он возвращает ссылку, медленный и в запросах не рекомендуется). Во-вторых, и это главное: измерение `ТипКлиента`, судя по данным (Юр, Физ), это перечисление или строка, а не ссылка на элемент справочника "Клиенты". То есть в параметр передаётся объект неверного типа, и запрос ничего не найдёт.

Как починить: передавать значение перечисления, например `Перечисления.ТипыКлиентов.Юридическое`, либо параметр того же типа, что и измерение регистра.

### 5. ТекущаяДата() вместо даты заказа

`Запрос.УстановитьПараметр("ДатаЗаказа", ТекущаяДата())` считает актуальность на "сейчас", а не на дату заказа. Для исторического или будущего заказа цена будет неверной. Плюс `ТекущаяДата()` берёт время сервера, а не сеанса, и для распределённых баз это вообще может быть не та дата.

Как починить: передавать реальную дату заказа, `Запрос.УстановитьПараметр("ДатаЗаказа", ДатаЗаказа)` (параметр функции).

### 6. Случай "вес вне интервалов" не возвращает NULL

По заданию вес за пределами максимального интервала означает, что стоимость не определена (NULL или 0). Код при пустом результате возвращает 0, и это смешивает "бесплатно" и "не определено". Ноль как валидная цена и ноль как "тарифа нет" становятся неразличимы.

Как починить: возвращать NULL (или явный признак "тариф не найден"), а решать, считать ли это ошибкой, должен вызывающий код.

### 7. Функция объявлена без параметров

`Функция РассчитатьСтоимость()`. Все входные данные (тип клиента, дата заказа) берутся из глобального контекста и из `ТекущаяДата()`, а не передаются. Из-за этого функцию нельзя переиспользовать и нельзя нормально протестировать (не прогнать юнит-тест с фиксированными входами).

Как починить: `Функция РассчитатьСтоимость(ТипКлиента, ДатаЗаказа, ТабличнаяЧасть)`.

### 8. Суммирование веса без проверки и обращение к неопределённой ТЧ

`Для каждого Строка из ТабличнаяЧасть.Номенклатура Цикл`: переменная `ТабличнаяЧасть` в области видимости не определена (нет ни параметра, ни объявления). А если табличная часть пустая, то СуммаВес = 0, и запрос начнёт искать интервал для нулевого веса. Случай "груза нет" стоит обработать явно (вернуть NULL или ошибку, а не искать тариф на 0 кг).

Как починить: передавать ТЧ параметром и проверять `Если СуммаВес = 0 Тогда Возврат NULL КонецЕсли;`.

### Итоговый исправленный вариант

```bsl
Функция РассчитатьСтоимость(ТипКлиента, ДатаЗаказа, ТабличнаяЧастьНоменклатура)
    СуммаВес = 0;
    Для Каждого Строка Из ТабличнаяЧастьНоменклатура Цикл
        СуммаВес = СуммаВес + Строка.Вес;
    КонецЦикла;

    Если СуммаВес = 0 Тогда
        Возврат NULL; // нет груза, стоимость не определена
    КонецЕсли;

    Запрос = Новый Запрос;
    Запрос.Текст =
    "ВЫБРАТЬ ПЕРВЫЕ 1
    |   Цены.Стоимость КАК Стоимость
    |ИЗ
    |   РегистрСведений.ЦеныДоставки.СрезПоследних(
    |       &ДатаЗаказа,
    |       ТипКлиента = &ТипКлиента
    |           И ВесОт < &СуммаВес
    |           И ВесДо >= &СуммаВес) КАК Цены
    |УПОРЯДОЧИТЬ ПО
    |   Цены.ВесДо ВОЗР";

    Запрос.УстановитьПараметр("ТипКлиента", ТипКлиента);
    Запрос.УстановитьПараметр("СуммаВес",   СуммаВес);
    Запрос.УстановитьПараметр("ДатаЗаказа", ДатаЗаказа);

    Результат = Запрос.Выполнить();
    Если Результат.Пустой() Тогда
        Возврат NULL; // вес вне известных интервалов
    КонецЕсли;

    Выборка = Результат.Выбрать();
    Выборка.Следующий();
    Возврат Выборка.Стоимость;
КонецФункции
```
